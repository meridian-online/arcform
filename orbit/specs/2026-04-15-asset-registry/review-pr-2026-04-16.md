## Findings

### [LOW] UNION/EXCEPT/INTERSECT queries not traversed for input discovery

**Category:** test gap / incomplete parsing
**Description:** `extract_inputs_from_query` calls `query.body.as_select()` which returns `None` for `SetOperation` bodies (UNION ALL, EXCEPT, INTERSECT). A SQL file like `SELECT * FROM a UNION ALL SELECT * FROM b` will discover zero inputs because the body is a `SetOperation`, not a `Select`. The same applies to CTAS with UNION: `CREATE TABLE t AS SELECT * FROM a UNION ALL SELECT * FROM b` will discover `t` as an output but miss `a` and `b` as inputs.
**Evidence:** `introspect.rs:124` -- `if let Some(ref body) = query.body.as_select()` silently drops non-Select bodies. No test for UNION queries exists.
**Recommendation:** Add a recursive arm for `SetExpr::SetOperation { left, right, .. }` that descends into both sides. Add a test for `CREATE TABLE t AS SELECT * FROM a UNION ALL SELECT * FROM b`. Not blocking for v0.2 because the graceful degradation design means the worst case is an undetected dependency, not a crash -- but worth fixing soon as UNION ALL is common in real DuckDB pipelines.

### [LOW] WHERE/HAVING subqueries not traversed for input discovery

**Category:** test gap / incomplete parsing
**Description:** `extract_inputs_from_query` only traverses FROM and JOIN clauses. Subqueries in WHERE (e.g., `WHERE id IN (SELECT id FROM blocklist)`) or HAVING clauses are not traversed, so `blocklist` would not appear as an input.
**Evidence:** `introspect.rs:123-131` -- only `body.from` and `table.joins` are visited. No traversal of `selection` (WHERE), `having`, or `group_by` expressions.
**Recommendation:** Consider using sqlparser's visitor trait (already enabled via `features = ["visitor"]` in Cargo.toml) to walk the full AST rather than hand-rolling traversal. This would catch subqueries in all positions. Not blocking for v0.2 -- WHERE subqueries referencing pipeline-internal tables are an edge case, and missed inputs merely skip validation rather than causing false positives.

### [LOW] Clippy warning in introspect.rs

**Category:** code quality
**Description:** `ref body` in a `Some` pattern creates a double reference where the compiler already borrows.
**Evidence:** `cargo clippy` reports `needless_borrow` at `introspect.rs:124`.
**Recommendation:** Change `if let Some(ref body)` to `if let Some(body)`. Trivial fix.

### [INFO] COPY TO semantics differ from spec wording

**Category:** spec-implementation tension
**Description:** AC-03 spec says "COPY ... TO are treated as the step producing/writing to those assets." The implementation correctly treats `COPY table TO file` as the table being an INPUT (read from), not an output. This is semantically correct -- COPY TO reads from a table -- but contradicts the spec's wording. The test `test_ac03_copy_to_file` asserts the table is in inputs, confirming the implementation is intentional.
**Evidence:** Spec AC-03 verification: "COPY target is handled (at minimum not erroring)" -- the test passes this bar. The file output is not tracked as an asset (no file asset model yet).
**Recommendation:** Clarify the spec wording for AC-03 in a future revision. The implementation is correct; the spec conflated "the step writes to a file" with "the step produces a table asset."

### [INFO] progress.md test count is stale

**Category:** documentation
**Description:** `progress.md` claims "55 total tests" but `cargo test` reports 56. The self-reference test (`test_self_reference_not_a_violation`) was added after the progress.md tally was written. This is cosmetic but could confuse future reviewers.
**Evidence:** `cargo test -- --list 2>&1 | tail -1` shows "56 tests, 0 benchmarks".
**Recommendation:** Update progress.md test count to 56.

### [INFO] No validation that `assets:` section references existing steps

**Category:** edge case
**Description:** If a user writes `assets: { foo: { produced_by: nonexistent_step } }`, the `AssetGraph::build` method generates a warning but does not error. The `Manifest::validate()` method does not check the `assets:` section at all.
**Evidence:** `asset.rs:107-112` -- warning is pushed but execution continues. `manifest.rs:104-148` -- no validation of `assets` map.
**Recommendation:** Consider adding a manifest validation check that `produced_by` references an existing step name. A warning-only approach is defensible for v0.2 (users may be iterating) but a hard error would catch typos earlier.

### [INFO] O(n^2) validation for large pipelines

**Category:** performance
**Description:** `validate_order` performs a linear scan of all steps for each read-asset to check `is_produced_by_any_step`. For a pipeline with S steps and R total read-assets, this is O(S * R). The producer lookup is also duplicated when a violation is found (the same linear scan runs twice). For typical pipelines (<100 steps) this is negligible.
**Evidence:** `asset.rs:146-150` -- `step_order.iter().any(...)` inside a loop over read assets.
**Recommendation:** No action needed for v0.2. If ArcForm grows to support 1000+ step pipelines, build a `HashMap<asset_name, step_name>` once upfront.

## Honest Assessment

This is a solid, well-structured v0.2 implementation. The three-layer architecture (infer, declare, override) is clean. Module boundaries are clear -- `introspect.rs` handles SQL parsing, `asset.rs` handles graph construction and validation, changes to existing modules are minimal. The 56 tests provide good coverage across all 10 ACs with integration tests in the runner that verify end-to-end behavior including the "no steps execute on validation failure" invariant.

The real gaps are in SQL traversal depth: UNION queries and WHERE subqueries are silently dropped during input discovery. These are not blocking because: (1) the graceful degradation design means missed inputs lead to missed validation, not false positives or crashes, and (2) users can always add explicit `depends_on` or `assets:` overrides. However, UNION ALL is very common in DuckDB pipelines (staging patterns, incremental loads), so this should be addressed in a follow-up. The `visitor` feature is already enabled in the sqlparser dependency, suggesting a full-AST walk was considered -- using it would close both gaps at once.

All v0.1 tests pass unchanged. The code is backwards-compatible. The one clippy warning is trivial. The spec-to-implementation alignment is strong, with only the COPY TO wording being slightly off. The prior inline review (2026-04-15) correctly identified the self-reference bug and the CTE alias edge case, both of which are handled.

Verdict: APPROVE. The UNION/WHERE gaps are worth tracking but not blocking for merge.
