## Findings

### [FIXED] Self-referencing step false positive
**Category:** bug
**Description:** A step that reads and writes the same table (e.g. `INSERT INTO t SELECT * FROM t WHERE ...`) would incorrectly trigger a dependency order violation if no prior step produces that table. The validation checked reads before recording the current step's produces.
**Fix:** Added a skip for self-references in `validate_order()` + test `test_self_reference_not_a_violation`.

### [INFO] CTE aliases appear as inputs
**Category:** edge-case
**Description:** CTE aliases (e.g. `WITH recent AS (...) SELECT * FROM recent`) appear in the inputs set. This is benign — since no step produces a CTE alias, it's treated as an external table and skipped during validation. No action needed.

### [INFO] COPY semantics differ between PostgreSQL and DuckDB
**Category:** edge-case
**Description:** sqlparser parses COPY using PostgreSQL semantics (COPY ... FROM STDIN / TO STDOUT). DuckDB's `COPY ... FROM 'file.csv'` may parse differently. The current handling is correct for the common `COPY table TO 'file'` case (tested). DuckDB's `COPY ... FROM` for file imports may need attention in a future iteration if users report issues.

## Honest Assessment

This is a clean, well-structured implementation of the asset registry. The three-layer approach (infer → declare → override) is implemented correctly with clear module boundaries. The 56 tests provide thorough coverage across all 10 ACs, including edge cases. The self-reference bug was caught and fixed during review. The implementation is backwards-compatible — all v0.1 tests pass without modification. The code adds ~1650 lines but the complexity is well-contained in `introspect.rs` and `asset.rs`, with minimal changes to existing modules.

The main limitation is that COPY and some DuckDB-specific syntax may not parse correctly in all cases, but the graceful degradation (AC-07) handles this by design — unparseable SQL warns and treats the step as opaque.
