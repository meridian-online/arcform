## Findings

### [CRITICAL] ac-04b hostile-tarball end-to-end test never exercises the extractor â€” panics in setup
**Category:** test-gap (verification-claim mismatch)

**Description:**
`src/registry/transport.rs:574-596 :: test_ac04b_hostile_tarball_extraction_rejected` is the test the spec calls out for ac-04b's "Unit test: build a hostile fixture tarball ... assert each hostile entry is rejected". The test panics on line 581 inside the `tar` crate's `Header::set_path`:

```
thread '...' panicked at src/registry/transport.rs:581:44:
called `Result::unwrap()` on an `Err` value: Custom { kind: Other, error: "paths in archives must not have `..` when setting path for " }
```

The `tar` crate refuses to *build* a tarball containing `..` segments, so the hostile archive is never produced and `GitTarballTransport::extract_tarball` is never called with a malicious entry. The validator `validate_tarball_entry` is unit-tested directly in 5 sibling tests and works correctly â€” so the protection is real â€” but the integration test the spec demands (extractor refuses a real hostile archive) does not run.

**Evidence:**
- `cargo test test_ac04b_hostile_tarball_extraction_rejected` â†’ 1 failed, 0 passed.
- File: `src/registry/transport.rs:574-596`.
- The benign-tarball companion (`test_ac04b_benign_tarball_extracts`, line 600) passes, confirming the extraction path itself works for safe inputs.

**Recommendation:**
Bypass the `tar` crate's safety check when building the hostile fixture so the hostile archive actually exists on the wire. Two options:

1. Use `tar::Builder::append_data` with a `Header` whose name is poked into the underlying byte buffer directly via `header.as_old_mut().name`, sidestepping `set_path`.
2. Hand-craft the hostile tarball bytes (POSIX ustar header is short â€” `name`, `size`, `chksum`, `typeflag`, two-byte header pad, data block padded to 512). A small fixture function in the test module is fine.

Either way the assertion remains: `GitTarballTransport::extract_tarball` returns `Error::RegistryTransport`, no file lands on disk, no `<dest>/` directory is left behind. Once that test passes the `cargo test` suite is fully green.
