## Findings

### [LOW] Undeclared --param keys are silently discarded
**Category:** edge-case
**Description:** `resolve_params()` iterates only over keys declared in the manifest's `params:` section. A `--param foo=bar` flag where `foo` is not a declared param is silently ignored — no env var is set, no warning is emitted. This could confuse users who misspell a param name (e.g. `--param star_date=2026-01-01` instead of `start_date`).
**Evidence:** `src/runner.rs` lines 49-68 — the loop is `for (name, param) in manifest_params`, so undeclared CLI keys in `cli_map` are never consumed.
**Recommendation:** Not blocking for v1. Consider emitting a warning (not an error) when a CLI param key doesn't match any declared manifest param. This preserves forwards-compatibility while helping users catch typos.

### [LOW] New dead-code warning for `run()` wrapper in bin target
**Category:** code-quality
**Description:** The backwards-compatible `run()` function (line 77 of runner.rs) generates a dead-code warning in the bin target because only `run_with_params()` is called from `cli.rs`. The function is used extensively by tests, so it compiles fine under `#[cfg(test)]`, but the bin target sees it as unused.
**Evidence:** `cargo check --all-targets` output: `warning: function 'run' is never used --> src/runner.rs:77:8`
**Recommendation:** Add `#[cfg(test)]` or `#[allow(dead_code)]` to the `run()` function, or make it `pub(crate)` with a `#[cfg(test)]` gate. Minor polish, not blocking.

### [INFO] Newline trimming matches bash $(…) semantics
**Category:** correctness-verification
**Description:** The spec review (v1) flagged ambiguity between "trim trailing newline" (singular) and bash `$(...)` which strips all trailing newlines. The implementation uses `trim_end_matches('\n')` which strips ALL trailing `\n` characters — this matches bash semantics. Verified correct.
**Evidence:** `src/engine.rs` line 130: `stdout_buf.trim_end_matches('\n').to_string()`

### [INFO] Backwards compatibility verified structurally
**Category:** correctness-verification
**Description:** All existing test helpers (in manifest.rs, asset.rs, runner.rs) were updated with `output: None`, `params: IndexMap::new()`, `dotenv: Vec::new()`, and match patterns were updated from `MockCall::Command { command }` to `MockCall::Command { command, .. }`. The `run()` wrapper delegates to `run_with_params(…, &[])` ensuring existing call sites pass an empty param list. All `#[serde(default)]` annotations ensure YAML without the new fields deserialises without error.

## Honest Assessment

This is a clean, well-scoped implementation that faithfully follows the spec's 10 ACs and 9 constraints. The architecture is sound: env vars as the parameter transport layer, strict ARC_PARAM_ namespacing, manifest-declared param contract, and output capture for command steps only. The Engine trait signature change (the highest-risk cross-cutting change) is handled correctly — both trait methods gain `env`, MockEngine records it, DuckDbEngine applies it via `Command::envs()`, and the backwards-compatible `run()` wrapper shields all existing tests.

The two LOW findings (silent discard of undeclared params, dead-code warning on `run()`) are minor UX and polish issues, not correctness problems. The implementation is ready to merge. The only thing preventing full test execution is the pre-existing `libduckdb.so` linker issue, which is a machine-specific environment problem documented in CLAUDE.md and not a regression from this branch.
