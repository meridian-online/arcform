## Findings

### [LOW] Pipeline timeout not checked between retry attempts

**Category:** edge-case
**Description:** The pipeline timeout check (`if elapsed >= pt`) fires once per step at the top of the outer loop (line 187-197), but not between retry attempts within the inner loop. If a step has many retries with non-zero backoff and the pipeline expires mid-retry-sequence, the step will keep retrying until exhaustion. Each retry gets a step_timeout clamped to `Duration::ZERO` (via `saturating_sub`), which for the real engine causes immediate timeout-then-retry cycles. This is wasteful but not incorrect — the step eventually fails.
**Evidence:** `src/runner.rs:187-197` (outer check) vs `src/runner.rs:235-311` (inner loop with no pipeline check before dispatch).
**Recommendation:** Acceptable for v0.1. The spec constraint says "Pipeline timeout checks fire before step execution, not during skipped steps" which this satisfies at the step level. Adding an inner-loop check between retries would be a quality-of-life improvement for a future iteration.

### [LOW] test_res_ac06_pipeline_timeout is non-deterministic

**Category:** test-gap
**Description:** The test acknowledges in its own comments that the mock may be too fast for the 0.001s pipeline timeout to fire: "If it succeeded, the mock was too fast — acceptable for CI." The test passes regardless of outcome (success is accepted silently). The companion test (`_deterministic`) partially compensates by exercising the timeout-fires path, though via step timeout rather than pure pipeline timeout.
**Evidence:** `src/runner.rs:2036-2043` — the `if let Err(e)` branch validates the error type, but success is silently accepted.
**Recommendation:** Acceptable. The deterministic companion test ensures the pipeline timeout path is exercised. A future improvement could use a `simulated_duration` field on MockEngine to reliably consume wall time.

### [LOW] total_retries not persisted to DuckDB state

**Category:** performance
**Description:** `DuckDbStateBackend::finish_run` accepts `total_retries` but discards it with `let _ = total_retries;`. This was flagged in v1 and acknowledged as intentional tech debt for v0.1.
**Evidence:** `src/state.rs:169-172`
**Recommendation:** Carry forward as tech debt. The trait/mock contract is correct and the value is observable in tests.
