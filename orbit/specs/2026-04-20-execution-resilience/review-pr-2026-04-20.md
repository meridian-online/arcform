## Findings

### [CRITICAL] StepTimeout is not retryable — violates spec constraint

**Category:** bug
**Description:** The spec constraint states: "Retry and timeout interact independently — a timed-out step counts as a failed attempt." This means a StepTimeout should be treated like a StepFailed: counted as a failed attempt and retried if attempts remain. However, in the implementation, StepTimeout falls into the catch-all `Err(e)` arm (runner.rs:296) which halts execution immediately without retrying.
**Evidence:** `src/runner.rs:296-301`:
```rust
Err(e) => {
    // Non-retryable errors (StepTimeout, StepExecution, etc.) — halt immediately.
    let _ = state.record_step(&step.name, &sql_hash, StepStatus::Failed);
    let _ = state.finish_run(&run_id, executed, "error", total_retries);
    return Err(e);
}
```
Spec constraint: "Retry and timeout interact independently — a timed-out step counts as a failed attempt"
**Recommendation:** Add an `Err(Error::StepTimeout { .. })` arm that stores the error in `last_error` and continues to the next attempt, identical to the `Err(Error::StepFailed { .. })` arm. The step timeout should also be recomputed per-attempt when pipeline timeout is active (see MODERATE finding below).

### [MODERATE] Step timeout is computed once, not per retry attempt

**Category:** bug
**Description:** `step_timeout` is computed before the retry loop (runner.rs:234-245). When pipeline timeout is active, the clamped value reflects remaining time at loop entry. After backoff sleeps and failed attempts consume wall clock time, subsequent retries use a stale (too generous) timeout value. This could allow a retry to exceed the pipeline deadline.
**Evidence:** `src/runner.rs:233-245` — computed once; `src/runner.rs:265-267` — reused for every attempt inside the `for attempt in 1..=max_attempts` loop.
**Recommendation:** Move the step_timeout computation inside the retry loop so it re-evaluates `pipeline_start.elapsed()` before each attempt. Alternatively, this is acceptable if the pipeline timeout check fires between retries (it currently does not — only fires before the first attempt of each step).

### [MODERATE] Missing test for ac-06 (pipeline timeout)

**Category:** test-gap
**Description:** There is no `test_res_ac06_*` test. The spec explicitly requires: "Unit test: manifest with very small pipeline timeout_sec; MockEngine configured with simulated_duration that exceeds remaining time; verify PipelineTimeout error names the step."
**Evidence:** `rg 'test_res_ac06' src/` returns no matches.
**Recommendation:** Add a test that sets `manifest.timeout_sec` to a small value (e.g., 0.001) and has MockEngine simulate enough delay (or just check that the pre-step pipeline timeout fires after a previous step consumes time). The MockEngine may need a `simulated_duration` field or a sleep-based approach.

### [MODERATE] Missing test for ac-09 (retry output separators)

**Category:** test-gap
**Description:** No `test_res_ac09_*` test exists. The spec verification says: "Unit test: verify MockEngine is called the expected number of times (confirms all attempts execute)." While other tests indirectly verify call counts, there is no dedicated ac-09 test that names itself as such.
**Evidence:** `rg 'test_res_ac09' src/` returns no matches.
**Recommendation:** Add `test_res_ac09_retry_separators` that exercises a retried step and verifies the expected number of engine calls. Even without capturing stderr output, the call count assertion is the spec's stated verification method.

### [LOW] test_res_ac03_retry_succeeds_on_third_attempt is misleading

**Category:** test-gap
**Description:** The test name claims to verify "succeed on third attempt" but the function body contains abandoned logic (`drop(engine)` twice), then only tests the "exhaustion with always-fail" scenario with max_attempts=2. It never actually tests succeeding on the third attempt.
**Evidence:** `src/runner.rs:1758-1812` — the function creates and drops two engines before testing a different scenario entirely. The "succeed on second attempt" test (test_res_ac03_retry_succeeds_on_second_attempt) covers the happy-path retry, but there is no actual test of succeeding on exactly the 3rd attempt.
**Recommendation:** Either rename the test to reflect what it actually tests (retry exhaustion), or implement the intended 3-attempt test using `set_fail_on_call` twice (which requires the mock to support failing on multiple specific call indices) or a counter-based approach.

### [LOW] total_retries not persisted to state

**Category:** performance
**Description:** The `finish_run` trait method accepts `total_retries: usize` but the DuckDB implementation discards it with `let _ = total_retries;`. The comment says "schema migration deferred." This is noted in the code and is technically spec-compliant (ac-08 only requires tracking, not persistence, and MockStateBackend does record it), but it means retry observability is only available during the test/mock path.
**Evidence:** `src/state.rs:170-172`:
```rust
// total_retries is tracked for observability but not stored in v0.1
// (schema migration deferred to avoid DuckDB ALTER TABLE complexity).
let _ = total_retries;
```
**Recommendation:** Acceptable for v0.1. Document as tech debt. The schema migration can happen when the next state change is needed.

## Honest Assessment

The implementation is structurally sound — it introduces retry, backoff, and timeout in a clean, backwards-compatible way across the Engine trait, manifest parsing, and runner execution loop. The code is well-organized and the test coverage is solid at 9/11 ACs. However, there is one critical spec violation: StepTimeout being treated as non-retryable directly contradicts the spec constraint that "a timed-out step counts as a failed attempt." This must be fixed before merge. The two missing tests (ac-06, ac-09) should also be added to maintain full traceability. The stale timeout computation inside the retry loop is a moderate concern that interacts with the critical finding — once StepTimeout becomes retryable, the stale timeout value becomes more consequential.
