## Findings

### [LOW] on_exit trigger condition is broader than spec states

**Category:** behaviour-divergence
**Description:** The spec states "on_exit always runs if on_init was attempted." The implementation uses `let should_run_exit = init_attempted || !manifest.steps.is_empty()` (runner.rs:424), meaning on_exit also runs when there's no on_init hook but steps exist. This is a reasonable extension — without it, a manifest with `on_exit` but no `on_init` would never trigger on_exit. However, it's technically broader than the spec's constraint.
**Evidence:** runner.rs line 424: `let should_run_exit = init_attempted || !manifest.steps.is_empty();`
Spec constraint 3: "on_exit always runs if on_init was attempted (try/finally semantics)"
**Recommendation:** No change needed. The broader condition is the correct behaviour for real-world use (users expect on_exit to fire after steps regardless of on_init presence). The spec's phrasing is narrower than intended — this is a spec documentation gap, not a code bug.

### [LOW] Inter-hook name collision not enforced

**Category:** validation-gap
**Description:** The validation checks hook names against pipeline step names (line 343: `seen_names.contains(&hook.name)`) but does not check for collisions between hooks themselves. Two hooks could have the same name (e.g., on_init and on_exit both named "cleanup") without triggering an error.
**Evidence:** manifest.rs lines 342-348 check against `seen_names` (populated from pipeline step names only). Hook names are never inserted into `seen_names`, so inter-hook collision is undetected.
**Recommendation:** Insert each validated hook name into `seen_names` after its check. This is a 1-line fix per hook but is a LOW priority since duplicate hook names cause no runtime harm (hooks are dispatched by phase, not name). Flagged in spec review v3 as acceptable.

### [LOW] Empty-steps early return bypasses hooks entirely

**Category:** edge-case
**Description:** runner.rs line 149 returns early with `Ok(())` when `manifest.steps.is_empty()`. A manifest with hooks but no steps would never execute any hooks. This is arguably correct (a pipeline with no steps is a no-op), and the spec's execution model assumes steps exist (constraint 2: "on_init -> steps[0..n] -> ..."). But it could surprise a user who defines only hooks.
**Evidence:** runner.rs lines 149-151: `if manifest.steps.is_empty() { ... return Ok(()); }`
**Recommendation:** No change for v1. The spec requires "Backwards compatible — manifests without hooks work identically" (constraint 10), and the existing behaviour for empty-step manifests is to print "No steps defined" and return. Changing this would alter pre-existing semantics. Document the limitation if it matters.

### [LOW] Duplicated env-var injection logic

**Category:** code-quality
**Description:** The failure context env var injection (`ARC_FAILED_STEP`, `ARC_EXIT_CODE`) is written twice — once for `on_failure` (lines 397-409) and once for `on_exit` (lines 435-449). The match arms are identical. A helper function would reduce duplication.
**Evidence:** runner.rs lines 397-409 and 435-449 contain identical match blocks.
**Recommendation:** Extract into a helper like `inject_failure_env(env: &mut HashMap, err: &Error)`. Non-blocking — this is a minor DRY improvement for a future refactor.

## Honest Assessment

This is a clean, well-structured implementation that faithfully follows the spec. The code is readable, the hook dispatch logic is straightforward, and the closure-based step loop extraction is a reasonable approach to capturing the result for hook routing. All 10 ACs are covered with tests that verify both positive and negative cases. The test infrastructure (MockEngine's `set_fail_on_call`) is reused effectively for hook-specific scenarios.

The four LOW findings are all non-blocking: the broadened on_exit trigger is arguably better than the spec's literal wording, inter-hook name collision is harmless, the empty-steps edge case is a deliberate pre-existing limitation, and the duplicated env-var logic is a future cleanup opportunity. No bugs, no spec violations, no missing test coverage.

The only constraint preventing full verification is the known libduckdb linker issue — tests compile but cannot link/run on this machine. `cargo check --all-targets` confirms compilation correctness. This is a pre-existing infrastructure limitation documented in CLAUDE.md.
