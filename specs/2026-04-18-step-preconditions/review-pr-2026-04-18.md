## Findings

### [LOW] ac-12 test covers adjacent behaviour, not the AC's primary concern

**Category:** test-gap
**Description:** AC-12 specifies "Precondition command execution error (binary not found, crash) halts the pipeline." The test `test_pre_ac12_command_nonzero_is_stale` verifies that `exit 1` returns `Ok(false)` (stale), which is the `Ok` path. The AC's actual concern is the `Err` path — when `sh` itself can't be spawned or the process crashes. The code handles this correctly (line 86-90 of precondition.rs returns `Err(PreconditionError {...})`) but the `Err` path isn't exercised by any test.
**Evidence:** precondition.rs:265 comment says "command non-zero exit = stale (not error)" — the test's own comment acknowledges it tests the non-error case.
**Recommendation:** Accept as-is. The `Err` path is hard to trigger in tests because `sh -c "nonexistent_binary"` successfully spawns `sh` which reports the error as a non-zero exit. Testing this would require making `sh` itself unavailable, which is fragile. The code is correct; the gap is cosmetic.

### [LOW] Untagged serde gives opaque errors for invalid precondition shapes

**Category:** edge-case
**Description:** `#[serde(untagged)]` on the Precondition enum means a YAML entry that matches neither `modified_after: {...}` nor `command: "..."` produces a generic serde "data did not match any variant" error rather than a clear "unknown precondition type" message.
**Evidence:** This is a known design trade-off chosen during the design interview — serde_yaml 0.9 requires `!tag` syntax for externally tagged enums, which conflicts with the desired YAML format. Untagged was the correct call.
**Recommendation:** Accept. Could add a custom deserializer in a future iteration if error messages prove confusing to users, but the validation step catches structurally invalid preconditions at load time.

### [LOW] Dead code warning for MockStateBackend::set_step_state

**Category:** minor
**Description:** `cargo test` produces a warning: `method set_step_state is never used`. This is a pre-existing test helper, not introduced by this PR.
**Evidence:** Compiler output during test run.
**Recommendation:** Out of scope for this PR. Could be cleaned up separately.

## Honest Assessment

This is a well-designed, well-tested feature addition. The typed precondition system is properly extensible — adding a new precondition type requires only a new enum variant and its `evaluate`/`validate` implementations. The `compute_staleness` refactor is minimal and reads clearly, with the `is_sql_hash_stale` extraction improving the original code's readability as a side effect. Test coverage is thorough: 28 new tests covering all four AND combinations for SQL+preconditions, both command precondition polarities, backwards compatibility, force override, and validation — all without sleep-based timing. The one gap (ac-12 execution error path) is genuinely hard to test and the code is correct. The implementation respects all 13 spec constraints. Ready to ship.
