## Findings

### [LOW] ALTER VIEW silently dropped
**Category:** Missing statement handler
**Description:** `ALTER VIEW my_view AS SELECT * FROM new_table;` parses successfully in sqlparser 0.55 (as `Statement::AlterView`) but falls through to the `_ => {}` catch-all in `extract_from_statement`. The view name is not captured in `outputs`, and the query's FROM tables are not captured in `inputs`. Both sets are returned empty.
**Evidence:** Edge probe test confirmed: `ALTER VIEW` parses to `Ok(SqlAssets { outputs: {}, inputs: {}, internal: {}, destroys: {} })`.
**Recommendation:** Add a `Statement::AlterView { name, query, .. }` arm analogous to the `AlterTable` handler. Low severity because ALTER VIEW is rare in data pipeline SQL, and it doesn't cause incorrect results (it degrades to an opaque step rather than producing wrong dependencies). Not blocking because the spec does not include an AC for ALTER VIEW.

### [INFO] AC-11 filtering happens at introspect layer, not validate_order
**Category:** Architecture observation
**Description:** The spec says "AssetGraph::validate_order ignores `internal` assets". In reality, `validate_order` has no awareness of `internal` at all. The CTE names never reach `reads` because `extract_assets` strips them from `inputs` before returning. The `internal` field on `StepAssets` is populated and available for downstream lineage (per constraint) but `validate_order` does not need to consult it.
**Evidence:** `validate_order` iterates over `step_assets.reads` only. CTE names were already removed from `inputs` (which maps to `reads`) by the `for cte_name in &assets.internal { assets.inputs.remove(cte_name); }` loop in `extract_assets()` (line 46-48 of introspect.rs).
**Recommendation:** This is correct behavior and arguably the cleaner design (filter early, not late). The test (`test_v03_ac11_cte_name_no_false_violation`) does verify the end-to-end property. No action needed -- just noting for architectural clarity.

### [INFO] DuckDB SUMMARIZE not supported by sqlparser
**Category:** Graceful degradation
**Description:** `SUMMARIZE orders;` fails to parse with sqlparser 0.55 DuckDB dialect, returning a parse error.
**Evidence:** Probe test: `Err(["sql parser error: Expected: an SQL statement, found: SUMMARIZE at Line: 1, Column: 1"])`.
**Recommendation:** This correctly triggers the opaque-step warning path. No action needed. Document as a known limitation if users report it.

### [INFO] MERGE with subquery source works correctly
**Category:** Edge case verified
**Description:** `MERGE INTO target USING (SELECT * FROM source1 JOIN source2 ON ...) AS src ON ...` correctly extracts `target` as output and both `source1` and `source2` as inputs, because the `source` field is a `TableFactor` and `extract_inputs_from_table_factor` handles `Derived` (subquery) variants recursively.
**Evidence:** Probe test: `Ok(SqlAssets { outputs: {"target"}, inputs: {"source1", "source2"}, internal: {}, destroys: {} })`.

### [INFO] CTE in subquery works correctly
**Category:** Edge case verified
**Description:** CTEs defined inside subqueries (not at top level) are correctly captured in `internal` and filtered from `inputs`, because `extract_inputs_from_query` is called recursively through `Derived` table factors.
**Evidence:** Probe test: `SELECT * FROM (WITH inner_cte AS (SELECT * FROM t2) SELECT * FROM inner_cte) sub;` produces `inputs={"t2"}, internal={"inner_cte"}`.

### [INFO] CREATE OR REPLACE VIEW works correctly
**Category:** Edge case verified
**Description:** `CREATE OR REPLACE VIEW foo AS SELECT * FROM bar;` correctly handled by the existing `Statement::CreateView` arm which already handles the `or_replace` flag.
**Evidence:** Probe test: `Ok(SqlAssets { outputs: {"foo"}, inputs: {"bar"} })`.

## Honest Assessment

This is a clean, well-structured implementation that meets all 14 acceptance criteria. The four-set SqlAssets model is a sound design: CTE filtering at the introspect layer (strip from inputs, collect in internal) is the right place to do it, preventing false dependencies from ever reaching the asset graph. Test coverage is thorough -- every AC has a named test, edge cases are covered (recursive CTEs, subquery CTEs, multi-table DROP), and the bonus tests add confidence. The only gap I found is the missing ALTER VIEW handler, which is a low-severity omission not covered by any AC and causes graceful degradation rather than incorrect behavior. Code quality is high: match arms are clear, the recursive traversal through extract_inputs_from_query/set_expr/table_factor handles nested structures correctly, and the CTE filtering loop at the end of extract_assets is simple and correct. Ship it.
